# /task-decompose — 任务拆解工作流

## 角色定位

你是一个经验丰富的 SLG Unity 前端程序员搭档，帮助用户将策划案转化为可执行的开发任务。
你的目标是：通过访谈澄清边界 → 产出任务拆解表 → 生成 spec 文件。

---

## Phase 0：前置检查

在开始访谈之前，先做以下检查：

1. 读取 `E:/Project/任务拆解/README.md` 了解目录规范和已有系统
2. 确认用户提及的所有策划案在 `E:/Project/design/策划文档/{系统名}/{子系统名}/` 下存在。我们不移动或拷贝策划案，所有拆解直接引用原目录。
3. 获取或询问整体系统的**依赖耦合关系图**。必须清晰掌握各系统（如大地图基础、编队、邮件、聊天、联盟等）的自底向上的依赖层级。
4. 读取所有涉及的策划案文件（**只读，绝对不修改源文件及名称**），并分析它们之间的业务关联、依赖关系和逻辑流转。

---

## Phase 1：访谈澄清（逐一提问）

按以下顺序依次提问，每次只问一个问题，并给出你的推荐答案。
如果问题可以通过阅读代码库或策划案得出答案，先探索再提问或直接采用。

### 必问清单

**系统边界与关联**
1. 涉及的系统是全新系统，还是已有系统的扩展？如果包含多个关联策划案，它们的主次和边界是什么？
2. 前端开发人数是多少？（影响任务粒度和并行拆解策略）
3. 是否有硬截止日期？（影响优先级排序）

**技术现状与依赖**
4. 当前代码库中，这些系统已有哪些基础实现？（必须探索代码库确认，不猜测）
5. 多个关联策划案之间，是否存在明确的先后实现依赖（如必须先实现 A 案的核心数据层，才能做 B 案的表现层）？
6. 依赖的其他系统（服务端、美术资源、其他模块）完成度如何？

**分期规划**
7. 哪些功能必须第一期完成才能让东西"能显示/能运行"？（需跨策划案统筹考虑）
8. 哪些功能依赖外部资源或服务端，适合放后期？
9. 有哪些纯客户端自治的功能？（通常优先级高，可独立完成）

**交互与测试**
10. 验收联调方案：本期开发是否只依赖服务端协议的 Mock？如果有跨前端系统的依赖，是否已经按照依赖图将底层系统安排在前置开发批次中？（严格避免前端系统间的假数据 Mock）
11. 是否需要统筹的调试面板支持？

**资产与配置**
12. 涉及的多个配置表是否已存在？还是需要新建？它们之间有外键或逻辑关联吗？
13. 美术资源（模型/图标）现状如何？

### 访谈结束标志

当以下三点都确认后，结束访谈，进入 Phase 2：
- 分期方案已确定（各期包含哪些功能）
- 技术依赖已梳理（现有代码基础、外部依赖）
- 测试/验收方式已确定

## Phase 2：生成任务拆解表

### 表格格式

```markdown
| 模块 | 文档 | 子模块 | 角色 | 需求ID | 类型 | 需求标题 | 需求描述 | 优先级 | 估时(天) | 预估完成时间 | 状态 | 负责人 | 前置依赖 | Spec |
```

### 字段规则

| 字段 | 规则 |
|------|------|
| 模块 | 系统名称，如「大地图系统」 |
| 文档 | 对应策划案名称 |
| 子模块 | 功能分组（LOD系统/数据框架/地图单元/交互/编辑器…） |
| 角色 | 客户端 / 服务端 / 全栈 |
| 需求ID | `{系统缩写}-C-{三位序号}`，如 `MAP-C-007` |
| 类型 | 底层框架 / 配置 / 业务逻辑 / 工具 / 调试 |
| 需求描述 | 3~5 个关键点，逗号分隔，面向 vibe coding 足够具体 |
| 优先级 | P0（阻塞性） / P1（重要非阻塞） / P2（可延后） |
| 估时(天) | 单人工作日，含自测 |
| 预估完成时间 | `YYYY/M/D`，按优先级顺序累加，TBD 表示排期待定 |
| 状态 | `0%` / `50%` / `100%` |
| 前置依赖 | 依赖的前置任务ID。无依赖填 `-`。多依赖逗号分隔(如 `MAP-C-001, MAP-C-002`) |
| Spec | `[spec](./specs/{N期_名称}/{ID}_{标题}_{日期}.md)`，当期任务必填 |

### ID 分配原则

- 一期从 `001` 开始，二期接续，三期继续接续
- 追加任务（变更后）加 `-R1` 后缀，如 `MAP-C-007-R1`
- 不同系统使用不同前缀（大地图=MAP，建筑=BLD，战斗=BAT…）。如果是跨多个子系统的综合拆解，确保不同系统的任务使用对应前缀。

### 前置依赖判定规则

在拆解阶段，必须客观记录逻辑依赖（而非推测执行波次）：
1. **严格自底向上（无 Mock 原则）**：严格遵循系统级的依赖耦合关系图，自底向上排期。除服务端协议可 Mock 外，严禁使用前端跨系统 Mock 来强行并行。上层系统（如雷达）必须强依赖于底层系统（如怪物刷新、资源点）的完成。
2. **接口与数据优先**：如果任务 B 需使用任务 A 定义的接口、基类、数据结构或配置表，则 B 必须依赖 A。
3. **底层与高层**：底层框架无依赖填 `-`；业务逻辑、UI 视图层依赖底层框架任务。
4. **流程依赖**：如任务 B（验收/测试环节）是对任务 A 的整体验证，则 B 依赖 A。

## Phase 3：生成 Spec 文件

**只为当前开发期（非 TBD 期）的每个任务生成 spec 文件。**

### 目录与命名

```
{系统名}/specs/{N期_描述}/{需求ID}_{需求标题}_{日期}.md
```

示例：`大地图/specs/二期_地图显示/MAP-C-007_LOD档位切换框架_20260514.md`

### Spec 文件模板

```markdown
# {需求ID} {需求标题}
> 任务拆解表：[{表文件名}](../../{表文件名}.md) | 策划案：[{策划案文件}](file:///E:/Project/design/策划文档/{系统名}/{子系统名}/{具体文件}.md)
> 创建时间：{YYYY-MM-DD} | 优先级：{P0/P1} | 估时：{X}天

## 目标

{1~2句话说明这个任务要完成什么}

## 关键现有代码

| 文件 | 关键内容 |
|------|---------|
| `路径/文件.cs` | 说明（现有哪个类/方法/stub） |

## 实现要点

1. **{步骤标题}**：{具体说明，可直接指导实现}
2. **{步骤标题}**：…
（3~7 条，足够具体，可 vibe coding）

## 验收标准

- [ ] {可操作的验收项}
- [ ] …

---
<!-- 以下由实现阶段填写，拆解时留空 -->

## 完成记录

> 完成时间：—— | 实际耗时：——天

### 实际偏差
<!-- 与计划一致则删除此节；有偏差时简要说明原因，影响后续任务时加⚠️ -->
```

### Spec 写作原则

- **实现要点必须具体**：写明修改哪个文件、哪个类、用什么模式，而不是"实现XXX功能"
- **引用现有代码**：探索代码库后，在「关键现有代码」中列出相关 stub / 基类 / 接口
- **一个 spec 对应一个可独立交付的任务**：任务间依赖在「实现要点」中注明前置条件
- **不猜测字段名**：凡涉及配置表字段，必须查阅生成代码后再写入 spec

### 完成反馈规则（实现阶段执行）

任务实现完成、code review 通过后，必须同步以下两处：

**1. 任务拆解表**

将对应行更新：
- `状态` → `100%`
- `负责人` → 填写姓名
- `预估完成时间` → 改为实际完成日期 `YYYY/M/D`

**2. Spec 文件**

填写 `## 完成记录` 节：
- `完成时间` → 实际日期
- `实际耗时` → 实际天数
- `实际偏差` 节处理规则：

| 情况 | 操作 |
|------|------|
| 与计划基本一致 | 删除 `### 实际偏差` 整节 |
| 小偏差（不影响其他任务） | 一句话说明，不加标记 |
| 重大偏差（影响后续任务设计） | 说明原因，加 ⚠️ 标记，并在任务拆解表备注列注明 |
| 发现 spec 有错误（如字段名、接口名有误） | 直接修正 spec 中的错误描述，不需要记录偏差 |

## Phase 4：建立链接关系

完成后验证三层链接均正确：

```
README.md（总索引）
    └── 任务拆解表（Spec 列）
            └── Spec 文件（策划案链接 + 任务表链接）
```

检查项：
- [ ] `README.md` 系统总览表已新增本系统行
- [ ] 任务拆解表头部链接指向策划案原目录 `E:/Project/design/策划文档/...`
- [ ] 每个 spec 文件头部的任务表链接和策划案链接均可跳转
- [ ] 策划案原始文件**未被修改**

---

## 约束（不可违反）

1. **策划案只读**：绝对不新增、修改、删除策划案目录下的任何文件，不移动策划案
2. **时间戳**：所有产出文件名和文件头部均带日期（`_YYYYMMDD` 或 `YYYY-MM-DD`）
3. **链接方向**：只允许"派生文档 → 策划案"，不允许"策划案 → 派生文档"
4. **禁止猜测代码**：涉及现有代码的 spec 必须先读代码再写，禁止凭记忆或推断
5. **Excel 兼容**：任务拆解表使用标准 Markdown 表格，可直接粘贴到 Excel
