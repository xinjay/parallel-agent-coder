# /start-task — 单任务实现工作流

用法：`/start-task MAP-C-007`

---

## Step 1：定位任务

1. 根据任务 ID 前缀（如 `FRM`、`MAP`），在 `任务拆解/` 目录下搜索所有 `任务拆解表_*.md` 文件：

   ```
   Glob: 任务拆解/**/任务拆解表_*.md
   Grep: <任务ID>   （在匹配到的拆解表中搜索）
   ```

   任何系统只需在 `任务拆解/` 下建目录并放入拆解表即可，无需注册映射。

2. 在任务拆解表中找到对应行，获取以下关键信息：
   - Spec 文件路径
   - 当前状态（若已是 100% 则提示用户确认是否重新实现）
   - 优先级、**执行波次 (Wave)** 和 **前置依赖**

3. **防呆拦截检查 (Wave Check)**：
   - 如果该任务属于 Wave N（N>1），主控**必须**扫描表格中所有该任务的「前置依赖」任务，检查它们的状态是否已达 `100%`。
   - 如果有前置任务未完成，**强制中断并警告用户**："*老大，当前任务 ({ID}) 属于 {Wave N}，但它的前置任务 (如 xxx) 尚未完成。强行开发可能会导致缺少基类或接口，确定要硬开吗？*"
   - 等待用户回复。如果用户坚持，才继续下一步。

4. 读取 spec 文件全文。

---

## Step 2：代码探索

根据 spec 的「关键现有代码」节，逐一读取相关文件。

**规则：**
- 只读 spec 中明确列出的文件，不做大范围扫描
- 如发现 spec 中的文件路径或类名有误，**直接修正 spec**，不中断流程
- 如发现 spec 遗漏了关键依赖文件，补充读取后继续

---

## Step 3：实现前确认

将以下信息呈现给用户，**等待确认后再动手写代码**：

```
任务：{需求ID} {需求标题}
优先级：{P0/P1} | 估时：{X}天 | 编排：{Wave N}

实现计划：
1. 修改 {文件A}：{做什么}
2. 新建 {文件B}：{做什么}
…

前置确认：
- [ ] 依赖任务 {ID} 已 100% 完成
- [ ] 相关配置表字段已查阅生成代码
- [ ] 通用性评估：该模块是否有复用价值？是否已提议将通用逻辑抽象为接口/基类？
```

用户回复「ok」或「可以」后进入实现。

---

## Step 4：实现

按 spec「实现要点」逐条执行。在实现前，**强制回顾并严格遵守当前项目的全局 AI 约定（如 `AGENTS.md`）**，严禁引入违反项目架构标准的设计。

**如果该任务属于特定领域（如 UI 拼包、配置表、战斗框架开发），必须先查阅并遵循该领域专属的专业 Skill（如 `slg-ui-edit`、`slg-config-edit` 等）。**

如果 Spec 中要求了单元测试，必须采用 TDD 方式（先写测试，后写实现）。

**强制检查点：**
- 异步操作使用 UniTask，不用 Coroutine/Task
- 新增 `MapUnitData` 派生类必须加 `[MapPoolable]` 特性
- Module 访问通过 ModuleContext，不用 GameContext
- 配置表字段名必须查阅 `Assets/Scripts/Configs/Gen/` 生成代码
- ViewExt 必须是 partial 类

**代码变动后：**
```
refresh_unity(compile="request") → read_console 确认无 error
```

---

## Step 5：验收核对（硬门禁）

**此步骤不可跳过、不可简化。「我感觉写完了」不等于验收通过。**

### 5a. 重读验收标准

重新打开 spec 文件，阅读 `## 验收标准` 节。每条 `- [ ]` 对应一项必须独立验证的标准。

### 5b. 逐条代码验证

对每条验收标准，**在代码中搜索对应实现**，确认代码真正满足条件：

1. 定位应满足此标准的代码
2. 用 Grep/Read 找到实际实现
3. 判断：代码是否完全满足标准？

| 判定 | Markdown |
|------|----------|
| 完整实现且正确 | `- [x]` |
| 部分实现或缺失 | `- [ ] 已知偏差: <具体缺什么>` |
| 无法静态验证（需运行时） | `- [ ] 待运行时验证: <原因>` |

**验证规则：**
- "代码存在"不等于"代码正确"——必须读完代码确认它做了标准要求的事
- TODO 注释、空方法体、硬编码占位值 = 未实现，标记已知偏差
- 标准说"4 种颜色"但代码只定义 2 种 = 已知偏差
- 标准说"不占队列"但代码无绕行逻辑 = 已知偏差
- 禁止对自己刚写的代码凭记忆打 `[x]`——必须回读文件确认

### 5c. 更新 Spec 验收标准

编辑 spec 文件 `## 验收标准` 节，将每条 `- [ ]` 替换为 5b 的判定结果。

### 5d. 填写完成记录

在 spec 末尾填写 `## 完成记录`：

```markdown
## 完成记录

> 完成时间：{YYYY-MM-DD} | 实际耗时：{N}天

### 实现概要
- {1-2 句话描述做了什么}

### 验收结果
- 通过: {N}/{total} | 已知偏差: {N} | 待运行时验证: {N}

### 已知偏差清单
- [ ] {标准}: {缺什么，什么原因，后续谁来跟进}
```

**已知偏差清单是 spec 文件中最重要的输出。** 它留在 spec 里，后续读者一眼就能看到什么没做完——无需重新审计。

### 5e. 计算完成百分比

```
完成百分比 = floor(通过数 / 总条数 × 100)
```

全部 `[x]` = 100%。4 条中 3 条通过 = 75%。

---

## Step 6：代码 Review

唤起 `agent-quality-checker` 子 Agent（或由主控亲自执行），触发 `slg-code-review` skill，完成两阶段审查：
1. spec 符合性审查
2. 代码质量审查（SLG 项目规范，包含架构与通用性评估）

如果 Review 发现问题，必须完成修复。Review 彻底通过后才能进入 Step 7。

---

## Step 7：完成反馈（必须执行）

### 7a. 更新任务拆解表

找到对应行，更新以下字段：
- `状态` → 更新为 Step 5e 计算的完成百分比
- `负责人` → 辛杰
- `预估完成时间` → 改为实际完成日期 `YYYY/M/D`

### 7b. 写回 Spec 文件

已在 Step 5c/5d 中完成。

偏差处理规则：

| 情况 | 操作 |
|------|------|
| 与计划基本一致 | 不记录偏差 |
| 小偏差（不影响其他任务） | 一句话说明 |
| 重大偏差（影响后续任务设计） | 说明原因，加 ⚠️，并在任务拆解表备注列注明 |
| 发现 spec 有错误 | 直接修正 spec 错误描述，不记录为偏差 |

---

## Step 8：AI 文档同步（按需触发）

任务完成且反馈写回后，若涉及核心逻辑或新机制，建议触发 `slg-doc-sync` skill：
- 分析本次任务的代码变更，更新关联的架构图、feature-map 或 system-overview。
- 如果本次任务沉淀了新的通用规则，可建议触发 `slg-rules-from-review` 更新工程约束。

---

## 约束

- **禁止跳过 Step 3 的确认**：未经确认不写代码
- **禁止跳过 Step 5**：验收核对是硬门禁，不是可选项
- **禁止跳过 Step 7**：完成反馈是工作流的一部分，不是可选项
- **禁止猜测字段名**：配置表字段必须查阅生成代码
- **策划案只读**：任何情况下不修改任何策划相关目录（如 `策划文档/`、`策划案/` 及其子目录）下的文件

## 常见跳过验收的借口

| 借口 | 现实 |
|------|------|
| "这任务太简单了，不用验" | 简单的任务验收也简单。30 秒的事。 |
| "我记得我实现了所有的" | 你记得的是你**想**实现的。grep 代码确认。 |
| "剩下那几条是小事" | 小事也是遗漏。标记已知偏差。 |
| "我等下回来验" | "等下"永远不会来。现在就验。 |
| "Spec 标准太模糊没法验" | 具体化解释，在完成记录中注明歧义。 |
| "FRM-C-014 我标了 100%" | 但实际上 4 个回调里 3 个是 TODO 桩。这就是验收核对要拦截的。 |

## 红灯信号 — 停下，核对

- "这个看起来做完了" → 重读 spec 验收标准
- "我记得这些都实现了" → grep 代码确认
- "剩下的是小问题" → 小问题也是问题，标记
- "我回头再验" → 回头不会来的，现在就验
- "Spec 不够细" → 具体化，在完成记录中注明
